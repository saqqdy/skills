# 🚀 Claude Code 插件方向建议（通用视角）

---

## 一、从 Claude Code 的独特优势出发

Claude Code 与普通工具的本质差异：**它能读代码、改代码、跑命令，并且有上下文记忆**。所以最好的插件方向应该满足：

> ✅ 需要 AI 判断力，不是规则脚本能搞定的
> ✅ 有明确的触发场景
> ✅ 输出可验证、可度量

---

## 二、6 个高价值方向

### 1️⃣ **Smart Code Reviewer** — 智能代码审查

**痛点**：人工 Code Review 质量参差不齐，很多 PR 只是走形式

**做什么**：
- PR 提交时自动触发，按维度打分：正确性、安全性、性能、可维护性
- 区分"必须修"和"建议修"，减少噪音
- 学习团队历史 review 模式，逐渐对齐团队风格

**为什么 AI 做**：理解业务语义，区分"这里 null check 是多余的"和"这里有隐含的空指针风险"，纯规则做不到

**市场**：每个有 PR 流程的团队

---

### 2️⃣ **CI Failure Triage** — CI 失败分诊

**痛点**：CI 红了，开发者花 20 分钟翻日志，发现就是个 flaky test 或者依赖超时

**做什么**：
- 读 CI 日志，判断根因类别：编译错误 / 测试失败 / 环境问题 / 超时 / 依赖冲突
- 对已知 flaky test 做指纹匹配，自动标记"这是已知 flaky，可重试"
- 对真正错误，给出最小修复建议

**为什么 AI 做**：日志千变万化，规则匹配覆盖不了，需要语义理解

**市场**：所有跑 CI 的团队，尤其是大型 monorepo

---

### 3️⃣ **Migration Copilot** — 技术迁移副驾驶

**痛点**：框架升级（Vue 2→3、Angular JS→Angular、Webpack→Vite、JS→TS）是所有团队迟早要面对的苦差事

**做什么**：
- 扫描项目，生成迁移影响报告（多少文件受影响、风险等级）
- 逐文件/逐模块生成迁移方案，人审核后一键应用
- 不能自动迁移的部分，标注原因和手动步骤

**为什么 AI 做**：迁移本质是语义转换，codemod 只能做语法级替换，AI 能理解"这段代码的意图"再做对应改写

**市场**：前端/全栈团队，中国市场 Vue 2 升级需求巨大

---

### 4️⃣ **Incident Commander** — 故障指挥官

**痛点**：线上出问题时，信息散落在监控、日志、部署记录、聊天里，人肉拼时间线效率极低

**做什么**：
- 自动收集多源数据：错误监控系统、日志平台、Git 变更、部署历史
- 生成事故时间线 + 根因分析 + 影响范围
- 输出结构化的 Post-Mortem 文档

**为什么 AI 做**：需要跨数据源关联推理，判断"部署 X → CPU 飙升 → 错误率上升"的因果关系

**市场**：SRE / On-call 工程师

---

### 5️⃣ **Prompt Engineer for Devs** — 开发者提示词工程师

**痛点**：大多数开发者不会写好 prompt，导致 AI 辅助编程效果差

**做什么**：
- 分析用户与 Claude Code 的对话历史，识别低效 prompt 模式
- 实时建议："你的问题太模糊，加点上下文效果会更好"
- 为常见开发任务预设高质量 prompt 模板（重构、调试、写测试等）

**为什么 AI 做**：评价 prompt 质量本身需要 AI 理解力

**市场**：所有 Claude Code / Cursor / Copilot 用户

---

### 6️⃣ **Doc Drift Detector** — 文档漂移检测器

**痛点**：代码改了文档没改，新人按旧文档操作就踩坑

**做什么**：
- 对比代码与文档（README、API 文档、注释），发现不一致
- 不只检查签名变化，还理解语义变化（"这个函数行为改了但文档没更新"）
- 自动生成文档更新草稿

**为什么 AI 做**：签名 diff 用脚本就行，但"行为变了"只有 AI 能判断

**市场**：开源项目、技术文档密集团队

## 四、排序建议

```
快速出成果 → CI Failure Triage（1-2 天可出 MVP）
长期最大价值 → Migration Copilot（可按框架拆分，逐个击破）
差异化最强 → Incident Commander（做多源关联推理的工具不多）
用户量最大 → Smart Code Reviewer + Prompt Engineer
```
