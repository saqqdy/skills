# 分析 Java 服务 Kafka 日志的 Skill 设计

## 一、需求分析

首先需要明确几个问题：

| 维度 | 需要确认的问题 |
|------|---------------|
| **日志来源** | Kafka 本身日志？还是 Java 服务发送到 Kafka 的业务日志？ |
| **日志格式** | JSON？纯文本？结构化？带 trace ID？ |
| **分析目标** | 故障排查？性能分析？业务监控？安全审计？ |
| **数据量** | 每天多少条？实时还是离线？ |

## 二、方向与方案对比

### 方向 1：日志采集层

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **Kafka Consumer API** | 实时、精准控制 | 需要写代码 | 实时监控、复杂逻辑 |
| **Kafka Connect** | 声明式、易扩展 | 配置复杂 | ETL、数据同步 |
| **Fluentd/Logstash** | 成熟生态、插件丰富 | 性能开销大 | 现有 ELK 栈 |
| **Kafka Streams** | 流处理、窗口聚合 | 学习曲线陡 | 复杂流分析 |

### 方向 2：日志解析层

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **正则表达式** | 简单快速 | 维护困难、易出错 | 固定格式 |
| **Grok 模式** | 可复用、可组合 | 性能一般 | Logstash 用户 |
| **JSON Schema** | 结构化、自描述 | 需日志源配合 | 现代日志系统 |
| **LLM 解析** | 智能、容错强 | 成本高、延迟大 | 非结构化日志 |

## 五、实现建议

### 推荐步骤

1. **第一步**：创建基础 Skill，只做日志格式识别和事件提取
2. **第二步**：添加参考文档（Kafka 模式库）
3. **第三步**：添加 Python 辅助脚本（解析、统计）
4. **第四步**：添加测试用例并迭代优化
5. **第五步**：添加 LLM 增强（智能诊断）

### 避免的坑

| 坑 | 解决方案 |
|-----|---------|
| 正则太复杂难维护 | 用 Grok 或 JSON Schema |
| 没考虑日志轮转 | 支持文件通配符和目录监控 |
| 分析结果不可操作 | 提供具体修复建议 |
| 性能问题 | 流式处理，批量聚合 |
